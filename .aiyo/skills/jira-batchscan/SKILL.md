# Jira Batch Scan — 批量 Jira Issue 扫描分析

## 启动交互流程

当用户调用此 skill 时，**必须**按以下步骤进行交互：

### Step 0: 选择分析维度

首先询问用户：

```
📋 Jira 批量扫描
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请选择分析维度：

1️⃣  按个人分析 - 分析某个团队成员的 Issues
2️⃣  按标签分析 - 分析特定标签的 Issues

请回复数字 1 或 2：
```

### Step 0a: 按个人分析

如果用户选择 **个人 (1)**：

```
👤 请输入要分析的团队成员用户名（如: shipeng.sun）：
```

等待用户输入用户名后，生成 JQL：
```jql
assignee = {{username}} AND status not in (Closed, Openlinux) ORDER BY updated DESC
```

### Step 0b: 按标签分析

如果用户选择 **标签 (2)**，显示预置标签选项：

```
🏷️  请选择要分析的标签：

1️⃣  Linux_security_drm    - Linux DRM 安全相关问题
2️⃣  android_security_drm  - Android DRM 安全相关问题

请回复数字 1 或 2（可多选，用逗号分隔）：
```

等待用户选择后，生成 JQL：
```jql
labels in ({{selected_labels}}) AND status not in (Closed, Openlinux) ORDER BY updated DESC
```

**预置标签参考**（参见 [references/config.md](references/config.md)）：
| 标签 | 说明 |
|------|------|
| `Linux_security_drm` | Linux DRM 安全相关问题 |
| `android_security_drm` | Android DRM 安全相关问题 |

### Step 0c: 确认搜索条件

在正式开始分析前，向用户确认：

```
🔍 搜索条件确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
JQL: {{generated_jql}}

预计分析 Issue 数量: {{count}}

是否开始分析？ (yes/no)
```

用户确认后，进入 Step 1。

## 核心原则

- **逐个分析**: 每次只处理一个 Issue，等待完成后再继续
- **不并行**: 禁止使用脚本循环或并行处理
- **自然语言激活**: 使用自然语言指令触发底层分析 skill

## 执行流程

### Step 1: 搜索 Issues

基于 Step 0 用户选择的维度，使用生成的 JQL 搜索 Issues。

**按标签搜索示例**：
```jql
labels in (android_security_drm) 
AND status not in (Closed, Openlinux) 
ORDER BY updated DESC
```

**按个人搜索示例**：
```jql
assignee = shipeng.sun 
AND status not in (Closed, Openlinux) 
ORDER BY updated DESC
```

**用户可额外指定**：
- 数量: "分析最近 50 个" → 设置 `MAX_RESULTS=50`
- 优先级: "只分析 P0/P1" → 添加 `AND priority in (P0, P1)`
- 时间范围: "最近一周的" → 添加 `AND updated >= -7d`

配置详情参见 [references/config.md](references/config.md)。

### Step 2: 逐个分析

对于每个 Issue，进行**简短总结和分析**（非深度分析）：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[1/N] ISSUE-KEY | 标题 | 状态
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**每个 Issue 的分析内容**：
1. **问题摘要** - 一句话概括核心问题
2. **当前状态** - 进展到哪一步，阻塞点是什么
3. **根因简析** - 简要分析可能的原因（基于已有信息，不下载日志）
4. **建议行动** - 下一步推荐动作
5. **优先级评估** - 是否需要优先处理

**分析格式示例**：
```markdown
### ISSUE-001: [Security] Widevine 授权失败

**问题摘要**: Android 设备播放 DRM 内容时出现授权失败错误。

**当前状态**: In Progress，已分配给开发团队，等待日志确认。

**根因简析**: 疑似 TEE 固件版本不兼容或密钥配置问题。

**建议行动**: 
- 收集完整 logcat 和 tombstone
- 确认设备固件版本

**优先级**: P1 - 客户报告，需要本周内响应
```

**进度跟踪**：
```
📊 进度：[15/50] (30%) | 成功：15 | 失败：0
```

## 执行注意事项

1. **简短分析**: 本 skill 专注于**快速扫描和总结**，不下载日志进行深度分析
2. **交互优先**: **必须**按 Step 0 的交互流程询问用户，不能直接执行
3. **进度显示**: 始终显示 `[x/N]` 进度，每 10 个输出一次摘要
4. **失败处理**: 单个 Issue 失败记录原因但继续，不中断整体流程
5. **模式识别**: 利用前几个 Issue 的结果，识别共性问题，调整后续分析重点
6. **标签验证**: 用户选择的标签必须在预置列表中，如不在需提示用户
7. **对话输出**: 报告**直接在对话中展示**，不自动保存本地文件。用户需要时再提供导出选项

### 示例 2: 按个人扫描
